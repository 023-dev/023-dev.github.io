## 컴파일러 방식

컴파일러 방식은 고급언어로 작성된 프로그램을 목적 프로그램으로 번환후 링킹작업을 통해 실행파일을 생성한다.
기계어 번역과정에서 많은 메모리를 사용한다.
초기 스캔시간은 오래 걸리지만 한번 실행 파일이 만들어지고 나면 실행속도가 빠르다.

## 인터프리터 방식

인터프리터 방식은 고급언어로 작성된 프로그램을 실행 시 한 번에 한 문장씩 기계어로 번역한다.
컴파일러와 같은 오브젝트 코드 생성과정이 없기 때문에 메모리 효율이 좋다.
변환속도는 빠르지만 한 번에 한문장씩 번역후 실행 시키기 때문에 실행 시간이 느리다.

## 그럼 자바는?

자바는 이 두가지 방식의 장점을 모두 활용하기 위해 컴파일러와 인터프리터 방식을 혼합한 형태로 동작한다.

1. 자바 컴파일러(Java Compiler)가 자바 소스파일을 컴파일한다. 이때 나오는 파일은 자바 바이트 코드(.class)파일로 아직 컴퓨터가 읽을 수 없는 자바 가상 머신이 이해할 수 있는 코드다. 
2. 컴파일된 바이트 코드를 JVM의 클래스로더(Class Loader)에게 전달한다.
3. 클래스 로더는 동적로딩(Dynamic Loading)을 통해 필요한 클래스들을 로딩 및 링크하여 런타임 데이터 영역(Runtime Data area), 즉 JVM의 메모리에 올린다.
   - 클래스 로더 세부 동작은 다음과 같다. 
   - 로드 : 클래스 파일을 가져와서 JVM의 메모리에 로드한다. 
   - 검증 : 자바 언어 명세(Java Language Specification) 및 JVM 명세에 명시된 대로 구성되어 있는지 검사한다. 
   - 준비 : 클래스가 필요로 하는 메모리를 할당한다. (필드, 메서드, 인터페이스 등등)
   - 해석 : 클래스의 상수 풀 내 모든 심볼릭 레퍼런스를 다이렉트 레퍼런스로 변경한다. 
   - 초기화 : 클래스 변수들을 적절한 값으로 초기화한다. (static 필드)
4. 실행엔진(Execution Engine)은 JVM 메모리에 올라온 바이트 코드들을 인터프리터 방식으로 명령어 단위로 하나씩 가져와서 실행한다. 

## 결국에 인터프리터면 느려지지 않을까?

C/C++ 언어는 소스코드를 컴파일하면 컴파일한 서버의 운영체제에 맞게, 기계어가 생성된다. 
즉, C/C++은 정적으로 컴파일을 하여 소스코드를 컴파일하는데 이렇게 정적 컴파일을 해주는 컴파일러를 AOT 컴파일러(Ahead-of-Time)라고 한다.
하지만  JIT 컴파일러는 앞서 살펴본 것처럼, 동적으로 코드를 컴파일하는 컴파일러를 의미한다.
JVM에서는 인터프리터의 단점을 보완하기 위해 JIT 컴파일러(Just-In-Time Compiler)을 사용한다.  
인터프리터로 실행되는 과정에서 프로파일러(Profiler)를 통해 자주 반복되는 코드(HotSpot)를 발견하면, 
해당 부분의 바이트코드를 JIT 컴파일러를 통해 네이티브 코드로 변환하고 캐싱된다.
이렇게 변환된 코드는 이후부터 인터프리터 모드보다 더욱 빠르게 실행된다.

> 프로파일이란 프로그램이 실행될 때 실행 시간, 메모리 사용량, 함수 호출 횟수 등과 같은 정보를 수집하고 분석하여 프로그램의 동작을 분석하는 과정을 말한다.
> 프로파일링을 통해 프로그램의 병목 지점을 찾고, 최적화활 부분을 결정할 수 있다.
> 프로파일러는 프로그램의 실행 정보를 수집하여 분석하는 도구로 자바에서는 VisualVM이 있다.

이렇게 프로파일 기반으로 런타임에 동적으로 컴파일을하면서 프로그램의 바이트 코드를 고도로 최적화한 기계어로 변환해준다.
단, 런타임 시에 컴파일되기에 바이트코드를 기계어로 변환하는 비용을 런타임에 지불하는 트레이드 오프를 해야한다.
